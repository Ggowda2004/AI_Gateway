ORIGINAL PROBLEM
================
Your /chat/ endpoint was returning 401 Unauthorized because API key validation 
was failing. Multiple issues combined to cause this failure.


ROOT CAUSES IDENTIFIED
======================

1. CHAT SCHEMA FIELD MISMATCH (PRIMARY ISSUE)
   -----------------------------------------------
   - Your schema defined: ChatRequest with field "msg"
   - Your endpoint tried to access: payload.message
   - Result: Pydantic validation error before auth was even reached
   - This was the FIRST problem that needed fixing


2. PREFIX LOGIC FAILURE (SECONDARY ISSUE)
   ----------------------------------------
   - You added a "key_prefix" column to optimize lookups
   - Migration set default value to empty string: ''
   - ALL EXISTING API KEYS had empty key_prefix
   - The validation logic tried to extract 16-char prefix from empty string
   - Result: Either skipped prefix match or fell back to all-keys check
   
   Database state:
   ✗ key_prefix field: '' (empty string for all existing keys)
   ✓ key_hash field: Properly populated with Argon2 hash


3. USER RELATIONSHIP NOT LOADED (TERTIARY ISSUE)
   -----------------------------------------------
   - validate_api_key() returned: key_record.user
   - In async SQLAlchemy, lazy-loaded relationships fail with greenlet error
   - User wasn't explicitly loaded in the query
   - Result: Either None or greenlet MissingGreenlet error
   
   Error would be:
   "sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called"


WAS THE PREFIX REALLY NEEDED?
=============================

Short answer: NO, it was not needed.

Explanation:
- The prefix optimization works ONLY if:
  1. ALL keys in database have the prefix populated
  2. You can guarantee new keys always have prefixes
  
- Problems in your case:
  1. Existing keys (created before prefix was added) have empty strings
  2. Migration didn't backfill existing keys with proper prefixes
  3. Mixed prefixed/non-prefixed keys break the optimization logic

- The prefix idea was good for performance, but:
  1. Not worth the complexity given your current key volume
  2. Requires proper migration strategy (backfill + not nullable)
  3. Adds failure points without corresponding benefit

async def validate_api_key(db:AsyncSession, key:str) -> User:
    """
    Validates an API key by verifying it against stored hashes in the database.
    Returns the User associated with the valid key.
    """
    if not key.startswith("kmgg_live_"):
        raise AuthError("Invalid Key or No key match found")
    
    try:
        # Query all active keys with user relationship eagerly loaded
        from sqlalchemy.orm import selectinload
        result = await db.execute(
            select(APIKey)
            .where(APIKey.is_active == True)
            .options(selectinload(APIKey.user))  # ← KEY FIX: Eager load!
        )
        active_keys = result.scalars().unique().all()  # ← Prevent duplicates
        
        # Verify the incoming key against each stored hash
        for key_record in active_keys:
            try:
                if verify_password(key, key_record.key_hash):
                    # Ensure user is loaded
                    user = key_record.user
                    if not user:
                        continue
                    return user  # ← User already loaded, no greenlet error!
            except Exception:
                continue
        
        raise AuthError("Invalid Key or No key match found")
        
    except AuthError:
        raise
    except Exception as e:
        raise AuthError(f"Error validating API key: {str(e)}")

What changed and why:
1. REMOVED prefix extraction logic
   → Because all keys have empty prefix anyway
   → Simplifies logic, fewer failure points
   
2. ADDED selectinload(APIKey.user)
   → Eagerly loads user in the same database query
   → Prevents lazy-load greenlet error in async context
   → User is guaranteed to be available
   
3. ADDED .unique() to scalars()
   → SQLAlchemy loads relationships by joining/repeating rows
   → .unique() deduplicates to prevent processing same key twice
   
4. SIMPLIFIED the verification loop
   → No prefix matching, just iterate active keys
   → Explicit null check for user relationship
   → Clean error handling

PERFORMANCE NOTE
================
Current approach: O(n) where n = number of active API keys
- Most systems: < 1000 keys = negligible performance cost
- Each verify_password() does ~100ms Argon2 verification
- Total time: Number of keys × 100ms, worst case ~100 keys = 10 seconds

If you later need optimization:
1. Backfill key_prefix for all existing keys
2. Make key_prefix non-nullable
3. Add unique constraint: (user_id, key_prefix)
4. Query by prefix first, then verify password on 1-2 candidates

LESSONS LEARNED

1. Don't add optimization before proving you need it

   - Added complexity without real benefit

   - Mixed state broke the optimization logic

3. Always test with realistic data

   - Failed in practice with empty values
   - Testing would have caught this immediately

4. Simpler solutions are often better
   - Removed prefix → validation still works perfectly
   - Fewer lines of code
   - Fewer failure points
   - Easier to understand and maintain
